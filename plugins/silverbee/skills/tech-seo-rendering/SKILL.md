# Skill: Tech SEO — Rendering & JavaScript

## Parallel Execution (MANDATORY)

Batch all independent tool calls into a single `run_multi_actions` call. When checking multiple URLs, fetching GSC + Ahrefs data, or querying multiple pages simultaneously, send all calls at once. Never call `run_action` sequentially for independent lookups — that is a performance bug.

## 1) Rendering Parity

Compare raw HTML (initial server response) against the fully rendered DOM. This is the foundational check — everything else builds on what this reveals.

**Method:** fetch the page as raw HTML (no JS execution), then fetch the rendered version (with JS execution). Diff the two.

**Checks:**
- Critical content (headings, body text, product data, pricing, reviews) present in raw HTML or only after JS execution.
- Navigation links: crawlable `<a href>` in raw HTML or generated by JS as non-crawlable elements (`<div>`, `<span>`, `<button>` with click handlers).
- Images: `<img>` tags with `src` in raw HTML or injected by JS. LCP image must not be lazy-loaded.
- Internal link targets: full URLs or relative paths in HTML, not JS route strings (`onClick` handlers, `router.push` calls without underlying `<a href>`).
- Meta tags and JSON-LD: present in raw HTML `<head>` or injected by JS after render. If JS-injected, flag as high risk — social platforms, AI crawlers, and non-Google bots read raw HTML only. *Full meta/canonical validation in `read_skill("tech-seo-indexation")`. Full schema validation in `read_skill("tech-seo-schema")`.*

**Classification per content element:**
- **Server-rendered:** present in raw HTML. Crawlable by all bots.
- **JS-dependent:** absent from raw HTML, appears after JS execution. At risk — Googlebot renders JS with delays; most other bots do not.
- **AJAX/fetch-dependent:** loaded via API calls after DOM ready. Highest risk — even Googlebot may not wait for secondary network requests.

**Output:**

| URL | Element | Raw HTML | Rendered DOM | Source (SSR / JS / AJAX) | Risk Level | Fix |

## 2) SSR Correctness Validation

**Run when the site uses SSR/SSG (Next.js, Nuxt, Gatsby, Remix, SvelteKit, or custom SSR).** A deployed SSR setup can fail silently.

**Checks:**
- **Content completeness:** does the server response contain actual page content, or an empty app shell (`<div id="root"></div>`) with client hydration expected to fill it? An empty shell means SSR is not functioning.
- **SSR/DOM match:** does server-rendered HTML match the client-rendered result? Mismatches cause hydration errors and inconsistent indexing.
- **Silent fallback:** some frameworks fall back to client-side rendering on SSR errors. Check whether the server response intermittently serves empty shells under load.
- **Cache freshness:** if CDN caching or ISR is used, check whether cached SSR output reflects current content. Stale markup (outdated prices, availability, text) is an indexing risk.
- **Error status codes:** when SSR fails, does the server return 200 with an empty shell (bad — Google indexes an empty page) or 500 (better — signals a problem)?

**Output:**

| URL | SSR Status | Shell or Full Content | SSR/DOM Match | Cache Freshness | Error Handling | Fix |

## 3) Dynamic Rendering & Bot Detection

Some sites serve pre-rendered HTML to bots and JS-rendered pages to users (via Rendertron, Puppeteer, prerender.io, or CDN edge workers).

**Checks:**
- Detect if the site serves different content based on user-agent. Fetch with a browser user-agent and with Googlebot user-agent — compare responses.
- If dynamic rendering is in use: verify the pre-rendered version matches the user version in content, links, and metadata. Significant differences = cloaking risk.
- Verify coverage extends to all important page types, not just the homepage.
- Check response headers for pre-rendering signals (`X-Prerender`, `X-Rendertron`, vary on user-agent).

Dynamic rendering is an acceptable workaround per Google's documentation. But if SSR is feasible, it is preferable — dynamic rendering adds complexity and a failure surface.

## 4) Hydration Failures

Hydration is when client-side JS attaches to server-rendered HTML. When it fails, content can disappear, revert to placeholders, or become non-interactive.

**Checks:**
- Console errors during page load (hydration mismatch warnings, JS runtime errors).
- Content placeholders or loading spinners that never resolve.
- Content visible in raw HTML that disappears or changes after JS executes (hydration overwrite).
- Interactive elements (forms, filters, navigation) that fail to become functional.
- Framework-specific patterns: React "Text content does not match", Vue "Hydration node mismatch", Next.js "Hydration failed".

**Output:** per-URL hydration issues with specific error messages, affected content, and fix.

## 5) SPA Routing

Single-page applications handle navigation client-side. This creates rendering-specific crawlability risks.

**Checks:**
- Each distinct content view must have a unique, crawlable URL (not hash fragments — `#/page` is not indexable).
- Direct URL access: each URL must return correct content when loaded directly by a bot, not redirect to the SPA shell. If the server returns the same HTML shell for every URL and relies on client JS to render the right content, bots see one page for every URL.
- Internal links must use `<a href>` tags. JS-only navigation (`router.push`, `onClick` without an anchor element) is invisible to crawlers.
- `<title>`, `<meta description>`, `<link rel="canonical">` must update on each route change in the rendered DOM. *Full canonical/meta validation in `read_skill("tech-seo-indexation")`.*

**Output:** SPA routing issues with specific URLs, affected elements, and fix.

## 6) JS Impact on Crawlability

JS performance (render-blocking, CWV, bundle optimization) is covered by `read_skill("tech-seo-performance")`. This section covers only the rendering-specific question: does JS prevent bots from accessing content?

**Checks:**
- Scripts that must execute before critical content renders — these delay bot access to content even when the bot does execute JS.
- Third-party scripts that inject or modify page content (A/B testing tools, personalization layers) — these can cause bots to see different content than users, or to see content only after significant delay.
- Client-side redirects via JS (`window.location`, `router.replace`) that bots may not follow — content at the destination URL is inaccessible if the redirect is JS-only.

*For full JS performance optimization, call `read_skill("tech-seo-performance")`.*

## 7) Guardrails

- Do not recommend a full SSR rewrite if a targeted fix solves the problem. Pre-rendering critical content sections or fixing hydration errors is cheaper and often sufficient.
- When SSR exists but is broken, fix the SSR pipeline first — do not layer dynamic rendering on top of a broken SSR setup.
- Content that must be crawlable belongs in raw HTML. If it can only exist via JS, the rendering strategy has a structural deficiency — flag it explicitly.
- Distinguish between Googlebot (supports most modern JS with delays) and other bots (Bing, AI crawlers, social platform scrapers — most do not execute JS). Recommendations must account for the full bot ecosystem.
- Every finding must reference the specific URL, content element, and rendering source. No generic "optimize your JavaScript" advice.
- Do not duplicate checks covered by other skills. Reference them when the rendering audit surfaces a related issue.
